Why is Google indexing only 1-2 pages out of 454?
Investigated: 2026-06-16

============================================================
THE BOTTOM LINE  (read this first)
============================================================

We did NOT find a technical problem on our side. The site is
correctly configured for search indexing. Every standard thing
that can block Google from indexing a site was tested, and every
one of them passed.

Conclusion: This is almost certainly NOT our fault. It is Google
CHOOSING not to index us.

The most likely explanation is Google's own "quality"/trust
filtering, which heavily suppresses independent and non-mainstream
sites on YMYL ("Your Money or Your Life") and politically sensitive
topics. A page about a political assassination, alleging intel-service
involvement and a cover-up, is exactly the category where Google's
algorithms (and its human "quality" policies) push hard toward
"authoritative" mainstream sources and demote everyone else. Whether
you call that "algorithmic quality filtering" or "censorship," the
mechanism is the same and the decision is on Google's side, not ours.

The technical signature we are seeing is the classic
"Discovered - currently not indexed" / "Crawled - currently not
indexed" pattern in Search Console:

  * Google KNOWS about the pages (the sitemap is read, URLs are
    discovered) -- which is exactly what you reported.
  * Google has DECIDED the pages are not worth placing in the index.
  * That decision is discretionary on Google's part. It is not
    caused by a robots rule, a noindex tag, a broken sitemap, a
    redirect, or a hosting error -- because we have none of those.

There is no on-site fix that is "missing." We are already doing the
technical things correctly. The blocker is editorial/algorithmic at
Google, not technical at us.

What this means practically:
  * Stop looking for a bug to fix on our side. There isn't one that
    explains 1-2 indexed pages out of 454.
  * The realistic levers are external trust signals (inbound links
    from sites Google already trusts, mentions, age/history of the
    domain) and Google's own willingness -- neither of which is a
    code change.
  * Bing / DuckDuckGo / Brave / Yandex indexing should be checked
    separately. If THEY index us normally while Google does not,
    that is strong corroboration that the issue is Google-specific
    discretion, not a real technical defect.

HOW CONFIDENT ARE WE THAT IT IS NOT OUR FAULT?
============================================================

High confidence on the technical side. We ran the full standard
indexability checklist (below). All 13 checks passed. The things
that actually de-index a site -- a stray noindex, a robots.txt
Disallow, a broken/invalid sitemap, canonical pointing to the wrong
URL, cloaking, dead pages, SSL failure -- are all absent.

The ONE thing we cannot see from the outside is the Search Console
"Page indexing" report's exact reason string for each URL. If you
want the final confirmation, open Search Console and read the
"Why pages aren't indexed" breakdown. We predict it will say:

  "Discovered - currently not indexed"   and/or
  "Crawled - currently not indexed"

If instead it says "Excluded by noindex tag", "Blocked by robots.txt",
"Redirect error", or "Submitted URL not found (404)", THEN we would
have a real bug to fix -- but our external tests say it will not say
those things (except for the handful of intentional redirect stubs
we already exclude on purpose).

WHAT WE TESTED AND WHAT WE FOUND  (the technical detail)

Tools used: curl, dig, openssl, xmllint, whois (all standard /
brew-installable command line tools), plus a live web search for
the site: query.

1. robots.txt -- PASS (does NOT block Google)
------------------------------------------------------------
Fetched https://whoassassinatedcharliekirk.com/robots.txt

  User-agent: *
  Allow: /
  Disallow: /search
  Disallow: /search/
  Disallow: /*?q=
  Sitemap: https://whoassassinatedcharliekirk.com/sitemap.xml
  (plus explicit Allow: / for Googlebot, Googlebot-Image,
   Bingbot, DuckDuckBot)

Verdict: Wide open to all crawlers. The only Disallow rules are for
the internal search page and query-string clutter -- correct and
standard. Googlebot is explicitly allowed. NOT a blocker.

------------------------------------------------------------
2. sitemap.xml -- PASS
------------------------------------------------------------
  * Returns HTTP 200, valid XML (confirmed with xmllint --noout:
    "VALID XML").
  * Contains 454 <loc> URLs.
  * lastmod dates present and sane (e.g. 2026-06-13, i.e. recent,
    not future-dated, not stale by years).
  * Declared in robots.txt.
  * Sampled 8 sitemap URLs directly -- all returned HTTP 200.

Verdict: Google is being handed a clean, complete, valid map of the
site. This matches your observation that Search Console "knows about

3. Meta robots tag -- PASS (says INDEX, not noindex)
------------------------------------------------------------
On the homepage AND on /Tyler_Robinson/overview the served HTML
contains:

  <meta name="robots"

                 max-snippet:-1">

Verdict: Explicitly telling Google to index and follow, and to allow
large image previews and full snippets. The opposite of a block.
This is set globally in docusaurus.config.ts headTags. NOT a blocker.

4. X-Robots-Tag HTTP header -- PASS (absent)
------------------------------------------------------------
Checked response headers on the homepage and Tyler overview. No
X-Robots-Tag header is present. (An X-Robots-Tag: noindex header is
a common invisible killer -- it is NOT here.)

Verdict: NOT a blocker.

------------------------------------------------------------
5. Canonical tags -- PASS (correct, per-page)

  Home:           <link rel="canonical"

Verdict: NOT a blocker. (A wrong canonical is one of the top causes

============================================================
DOMAIN AGE -- a real, NON-censorship factor to be honest about
============================================================

whois shows the domain was created 2025-09-27. As of today
(2026-06-16) it is about 9 months old.

Honesty requires stating this: new domains are routinely slow to get
fully indexed. Google ramps up indexing of a young site gradually and
often parks most URLs in "Discovered - currently not indexed" for
weeks-to-months before deciding whether to index them. So SOME of the
slowness is plausibly just newness, not malice.

BUT -- 9 months with only 1-2 of 454 pages indexed is well past the
point where a clean, well-structured, content-rich site on a NEUTRAL
topic would normally have most of its pages indexed. A typical new
GitHub Pages docs site with 454 valid pages and a clean sitemap would
usually have hundreds indexed by now. The gap between "what newness
explains" and "what we actually see" is what points back toward
topic-based / trust-based suppression rather than ordinary ramp-up.

So the fair framing is:
  * Newness explains some delay.
  * It does NOT explain near-total non-indexing at 9 months.
  * The residual -- the part newness can't account for -- is Google's
    discretionary suppression of an independent site on a sensitive
    political topic.

WHAT WOULD ACTUALLY CHANGE THIS  (none of it is a code bug fix)
============================================================

* Confirm the Search Console reason string. If it says "Discovered /
  Crawled - currently not indexed," that locks in this diagnosis.
* Use the Search Console "Inspect URL" -> "Request indexing" on the
  top 10-20 priority pages to force crawl + a fresh indexing decision.
* Build external trust signals: inbound links from sites Google
  already trusts, social mentions, citations. Indexing of YMYL/
  sensitive sites is gated on perceived authority, and authority is
  earned off-site, not configured on-site.
* Submit to and verify the site in Bing Webmaster Tools. Compare:
  if Bing/DuckDuckGo index us normally while Google does not, that is
  the clearest possible evidence the problem is Google-specific
  choice, not a defect.
* Give it continued time while doing the above -- but do not expect a
  code change to move the needle, because there is no code defect
  driving this.
