============================
GOAL
============================

Give every image page under {PHOTOS_DIR} a "Next Image" button directly beneath
its image, so a visitor can walk the ENTIRE image corpus one picture at a time
without ever going back to a table of contents. The button is a dark blue
({BTN_FILL}) filled pill with white text reading "Next Image" and a white right
chevron. Clicking it loads the NEXT image page in a single global traversal of
the whole hierarchy, and the traversal LOOPS — the last image's button returns
to the first image, so the walk never dead-ends.

Two things this prompt produces, and it re-produces them every run because the
YAML changes between runs:

  1. A new per-image property, next_image, written INTO {HIERARCHY_FILE}. Its
     value is the image_page (the full ~-rooted path) of the image that comes
     next in the global traversal. This is bookkeeping the button reads from —
     it is computed here, persisted here, and refreshed here.

  2. On every image page (the single-image leaf page the YAML binds as
     image_page — what the site model calls a Level 5 page because it shows one
     image), the "Next Image" button rendered under the image, its href pointing
     at the site-relative URL of that image's next_image. In the same pass the
     image itself is re-embedded correctly: the right served src and the right
     hyperlinks, because both can drift between runs.

This prompt is deterministic and mechanical. It does NOT write prose, does NOT
read the whole {CK_FILE} for meaning, and does NOT create, move, or delete
pages — every image page it touches already exists (it is generated by
{IMAGE_PAGE_PROMPT}). It writes one YAML property and refreshes one button plus
the image embed on pages that are already there. Run {IMAGE_PAGE_PROMPT} first
if pages are missing; this prompt assumes they exist.

It is idempotent: same YAML in, same next_image values and same buttons out.

============================
KNOWLEDGE — THE TRAVERSAL ORDER (WHAT "NEXT" MEANS)
============================

There is exactly one global sequence of image pages, S = [p0, p1, ... p(N-1)],
built by walking {HIERARCHY_FILE} once in DOCUMENT ORDER, depth-first,
pre-order:

  * Visit the level_3 nodes in the order they appear in the file.
  * At every cluster node (level_3, level_4, or level_5), in this order:
      (a) emit that node's OWN image pages, in the YAML order of its images
          list, THEN
      (b) descend into each of its child clusters in YAML order, applying this
          same rule to each.
  * Concatenate every image page emitted, across the whole tree, into the one
    flat sequence S.

next_image is then the successor in S, and the chain closes into a loop:

    next_image(p_i)       = image_page(p_(i+1))     for 0 <= i < N-1
    next_image(p_(N-1))   = image_page(p_0)          the last wraps to the first

Worked example (the shape the request described). FBI is a level_3. Under it
are level_4 groups A, B, C, D. Suppose FBI itself holds no own images and Group
A holds five images:

    A#1 -> A#2 -> A#3 -> A#4 -> A#5 -> B#1 -> B#2 -> ... -> D#last

  * A#1's next_image is A#2's image_page; A#5's next_image is B#1's image_page —
    the walk leaves Group A only after all of Group A is seen.
  * D#last is the last image under the FBI subtree, so its next_image jumps to
    the FIRST image of the NEXT level_3 in the file — a different part of the
    tree entirely.
  * If FBI also held its own images (real FBI does: 21 own, more in children),
    those own images come FIRST, before Group A — rule (a) before rule (b).

The very last image page in the entire file wraps back to the very first image
page in the entire file. There is always a next; there is never a dead end.

What is and is NOT in S:

  * S contains only image pages that actually exist and are publishable. An
    image that is BANNED (in the ban set) or that types as VIDEO / UNKNOWN has
    NO image page (see the two gates below); it is skipped entirely and is NOT
    a node in S. The chain steps over it — the image before it points at the
    image after it.
  * Cluster overview pages (the level_3/4/5 overview.mdx navigation pages) are
    NOT in S. The button walks pictures, not tables of contents.
  * A single-image node that is BYPASSED in navigation (its overview page has no
    inbound links) still contributes its one image page to S in document order,
    exactly where it sits in the tree. Bypass is a navigation-link rule; it does
    not remove the image from the global walk.

Consistency with the existing per-cluster prev/next: {GEN_SCRIPT} threads a
"previous/next within this cluster" sequence used in the "More In This Cluster"
list on each image page. That per-cluster list stays. The "Next Image" button is
the GLOBAL chain and is a separate, additional control — it keeps going after
the cluster ends, where the per-cluster "Next" stops. Keep both; do not
conflate them.
