============================
GOAL
============================

Give every video page under {VIDEOS_L2_DIR} a "Next Video" button directly
beneath its player, so a visitor can walk the ENTIRE video corpus one clip at a
time without ever going back to a table of contents. The button is a dark blue

chevron. Clicking it loads the NEXT video page in a single global traversal of
the whole hierarchy, and the traversal LOOPS — the last video's button returns
to the first video, so the walk never dead-ends.

Two things this prompt produces, and it re-produces them every run because the
YAML changes between runs:

     video_page — what the site model calls a Level 5 page because it shows one

============================
KNOWLEDGE — THE TRAVERSAL ORDER (WHAT "NEXT" MEANS)
============================

There is exactly one global sequence of video pages, S = [p0, p1, ... p(N-1)],
built by walking {HIERARCHY_FILE} once in DOCUMENT ORDER, depth-first,
pre-order:

  * Visit the level_3 nodes in the order they appear in the file.
  * At every cluster node (level_3, level_4, level_5, or deeper), in this order:
      (a) emit that node's OWN video pages, in the YAML order of its videos
          list, THEN
      (b) descend into each of its child clusters in YAML order, applying this
          same rule to each.
  * Concatenate every video page emitted, across the whole tree, into the one
    flat sequence S.

next_video is then the successor in S, and the chain closes into a loop:

    next_video(p_i)       = video_page(p_(i+1))     for 0 <= i < N-1
    next_video(p_(N-1))   = video_page(p_0)          the last wraps to the first

Worked example. FBI is a level_3. Under it are level_4 groups A, B, C, D.
Suppose FBI itself holds no own videos and Group A holds five videos:

    A#1 -> A#2 -> A#3 -> A#4 -> A#5 -> B#1 -> B#2 -> ... -> D#last

  * A#1's next_video is A#2's video_page; A#5's next_video is B#1's video_page —
    the walk leaves Group A only after all of Group A is seen.
  * D#last is the last video under the FBI subtree, so its next_video jumps to
    the FIRST video of the NEXT level_3 in the file — a different part of the
    tree entirely.
  * If FBI also held its own videos, those own videos come FIRST, before Group A
    — rule (a) before rule (b).

The very last video page in the entire file wraps back to the very first video
page in the entire file. There is always a next; there is never a dead end.

What is and is NOT in S:

  * S contains only video pages that ACTUALLY EXIST on disk under {VIDEOS_L2_DIR}
    and are publishable. A video whose video_page is "" (no page written yet) is
    NOT in S — the chain steps over it and it is reported so {VIDEO_PAGE_PROMPT}
    can write it. A BANNED video has no page, so it is not in S either.
  * A MEDIA-PENDING page IS in S. A video with cid "" or ipfs_pinned false still
    has a full Level 5 page — title, write-up, sources, and an honest
    "media pending" note where the player would be. That page is a real
    destination worth reaching, so it is a node in S like any other. The button
    on it still points onward; the button INTO it still lands on a real page.
  * Cluster overview pages (the level_3/4/5 overview.mdx navigation pages) are
    NOT in S. The button walks clips, not tables of contents.
  * A single-video node that is BYPASSED in navigation (its overview page has no
    inbound links) still contributes its one video page to S in document order,
    exactly where it sits in the tree. Bypass is a navigation-link rule; it does
    not remove the video from the global walk.
  * The two PROTECTED pages under {VIDEOS_L2_DIR} — overview.mdx and
    buckley-carlson-kash-patel-valhalla.mdx — are handled by their own rule:
    overview.mdx is a cluster page and is never in S. The hand-written
    buckley-carlson page predates this pipeline and carries no ck_ frontmatter;
    it is NOT woven into S and gets NO button (leave its body untouched). If its
    video also appears in the YAML with its own generated page, that generated
    page is the S node; record the collision in {FINDINGS_FILE} and let a human
    decide, exactly as the sibling prompts do.
